# Sparks — Screen Specifications

## Global rules

- App name: Sparks (lowercase "sparks" in logo)
- Stack: Next.js 14 App Router, TypeScript, Tailwind, shadcn/ui, Framer Motion
- All routes under (app) require an authenticated session
- All routes under (marketing) are public
- Generous whitespace, 0.5px borders, rounded-xl cards
- Color palette uses Tailwind's stone-* for grays, plus custom sage, terracotta,
  amber, cream defined in tailwind.config.ts

## Top nav — logged-out
Logo · How it works · Brands · Sign in (button, top-right)

## Top nav — logged-in
Logo · Today (active state) · Saved · Search · Avatar circle (right)

## Screen 1: Landing (route: /)

Purpose: sell the value prop; drive Pinterest signup.

Layout:
- Top nav (logged-out variant)
- Hero, max-width 480px, centered
  - Eyebrow: "Modest fashion, finally findable"
  - H1 (32px, weight 500): "Clothes that fit your style *and* your standards."
    The word "and" is wrapped in `<em class="font-serif">`
  - Body: "Tell us where your boundaries are. We'll search hundreds of brands,
    read the reviews, and bring back only what actually fits."
  - Primary CTA: "Continue with Pinterest" → /api/auth/pinterest/start
  - Secondary CTA: "See how it works" → /how-it-works
  - Caption under CTAs: "Pinterest OAuth · we never post or share"
- Three-step strip in footer: Connect / Refine / Discover

For the demo build: skip /how-it-works and /brands routes. They can return a
simple placeholder page.

## Screen 3: Select board (route: /onboarding/select-board)

Purpose: user picks the Pinterest board to use as their style baseline.

After OAuth, fetch user's boards. Render a grid of board cards (cover image,
name, pin count). User picks one. Show loading state ("Reading 47 pins…")
while pulling pins via /api/pinterest/pins.

For the demo build: live-fetch boards via Pinterest API. After selection,
pull up to 25 pins (not 100 — keep it fast).

## Screen 4: Style summary (route: /onboarding/style-summary)

Purpose: the trust moment — show the AI "gets" the user.

Layout:
- 4-step progress bar (step 2 active)
- H2: "Here's what we learned."
- Pull quote in serif italic, 18px, with 2px left border accent. Background
  is stone-50, padding 16px 18px. Content is generated by GPT-4o from
  the analyzed pins. Example: "You lean earthy and romantic — flowing midi
  silhouettes, muted greens and creams, a soft cottagecore through-line with
  structured tailoring on the edges."
- Tag chips with confidence percentages, color-coded:
  - >80% confidence: sage background (#EAF3DE), dark green text (#173404)
  - 60-80%: cream background (#FAEEDA), brown text (#412402)
  - <60%: stone-100 background, stone-600 text
- 6-tile color swatch grid showing dominant palette pulled from pins
- Caption: "Pulled from N pins on '[board name]'"
- Two CTAs: "Looks right →" (primary, advances to swipe) and "Not quite" (text)

"Not quite" behavior:
- Original summary fades to 55% opacity
- Correction panel expands below it with:
  - H3: "What did we get wrong?"
  - Body: "No need to rewrite the whole thing — just tell us what's off."
  - Textarea (min-height 56px) with placeholder example
  - Label "Or tap a quick fix:" then 5 chips:
    - "Less feminine" / "More minimal" / "More tailored" /
      "Bolder colors" / "Different season"
  - Selected chips toggle filled state (sage bg, sage text)
  - "Try again" (primary) and "Cancel" (text) buttons
  - Footer caption: "The swipe quiz is next — that'll catch anything we
    still get wrong."
- "Try again" POSTs to /api/style/refine, animates new summary in
- "Cancel" collapses the panel

For the demo build: GPT-4o call is real. Vision analysis on pins is
real. The "Not quite" correction flow is REAL — it's a strong demo moment.

System prompt addition for warmer voice (since GPT-4o defaults to clinical):
"Write in a warm, observational tone — like a friend who's known the user for
years describing their style. Avoid clinical or fashion-industry jargon. Use
unexpected adjective pairings ('softly structured,' 'quietly bold') sparingly
to make the description feel specific to this person rather than generic."

## Screen 5: Swipe quiz (route: /onboarding/swipe)

Purpose: refine aesthetic profile, gamified.

Layout:
- 4-step progress bar (step 3 active)
- H2: "Would you actually wear this?"
- Counter top-right: "12 / 25"
- Card stack: 3 visible cards stacked with slight rotation, top draggable
- Cards show full-bleed outfit image with bottom strip listing tags
- Swipe gestures via Framer Motion's drag:
  - Drag right > 120px → "LIKE" (sage stamp visible during drag)
  - Drag left > 120px → "NOPE" (terracotta stamp)
  - Drag up > 120px → "LOVE" (amber stamp)
- Three button row below cards (skip / love / like) for accessibility
- Caption: "Swipe left to skip · right to like · up to love"

Behavior:
- Each swipe writes to swipes table (or local state for demo)
- After 25 swipes, advance to /onboarding/standards

For the demo build: hardcode 25 outfit images from a static array (use a mix
of chic and gothic items pulled from products.json image_urls). Swipes save
to local state, no DB write needed. Skip the adaptive logic — just show in
order. If time runs short, this entire screen can be cut: button at the
bottom of style summary becomes "Skip quiz, set standards →".

## Screen 6: Standards (route: /onboarding/standards)

Purpose: capture user's modesty standards in their own words.

Layout:
- 4-step progress bar (step 4 active)
- Eyebrow: "Last step"
- H2: "Tell us what's off-limits."
- Body: "Describe your standards in your own words — neckline, sleeve length,
  fabric, fit, anything. We'll turn it into rules and enforce them on every
  search. You can edit this later."
- Single textarea, min-height 100px, placeholder example
- Subtle examples card (stone-50 background) showing 3 sample phrasings:
  - "Knees covered when sitting, sleeves past elbows, no cleavage."
  - "Long sleeves and ankle length, no fabric clinging at the waist."
  - "Just nothing transparent or backless. Otherwise I'm easy."
- CTA: "Find my style →"

Behavior:
- Submit POSTs free text to /api/standards/parse
- GPT-4o returns structured JSON with parsed boundaries
- Save to standards table (or local state for demo)
- Advance to /onboarding/confirm

For the demo build: GPT-4o call is real. Use OpenAI's structured outputs /
JSON mode to guarantee valid JSON. This is one of the strongest "AI is
reading my plain English" moments and it should work for real.

---

## Screen 7: Confirm parsed standards (route: /onboarding/confirm)

Purpose: show user how their text was interpreted; let them adjust.

Layout:
- Display parsed boundaries as readable bullet list:
  - "Sleeves: at least to the elbow"
  - "Hem: at or below the knee"
  - "Fabric: must be opaque (lined or thick)"
  - etc.
- Each line has small "edit" link
- Bottom CTAs: "Looks right →" (advances to /today) and "Let me rewrite"

For the demo build: render the structured JSON returned from /api/standards/parse
as readable bullets. Edit/rewrite can be skipped — just "Looks right" works.

## Screen 9: Search / Chat (route: /search)

Purpose: the magic moment — agent runs visibly.

Layout:
- Search bar at top: "I need a summer wedding guest dress, under $150…"
- On submit, thinking sequence appears as a vertical stack of step cards:
  1. "Read your request" → checkmark + parsed details
  2. "Searched 47 brands" → "Found 312 dresses matching basic criteria"
  3. "Applied your boundaries" → "Dropped 247 — sleeveless, deep V, hem too short"
  4. "Reading reviews for transparency…" → spinning state
  5. "Ranking against your style" → checkmark
  6. "Writing your picks" → checkmark
- Each step appears with 600-1000ms delay between them
- Caption at bottom: "~5 seconds · we never browse, we curate"
- After thinking completes, results appear below:
  - Compact summary bar: "Found 9 picks from 312 candidates."
  - Sub-line: "Skipped 5 with see-through warnings · 1 marked 'needs slip'"
  - Grid of 2-column product cards with full verification details

Product card on results page contains:
- Image (4:5 aspect ratio)
- Brand label (uppercase tiny) and price (top right, font-weight 500)
- Title
- "Why we picked this" line in serif italic, stone-50 background, padding
- Bulleted check list, separated by 0.5px top border:
  - Green checks (#0F6E56 text) for confirmed criteria
  - Amber warnings (#BA7517 text) with "!" for review-flagged concerns
- "View at [Brand]" CTA button (primary fill) + heart save button

For the demo build:
- The thinking sequence is REAL TIMING but with FAKED steps. The numbers
  ("47 brands", "312 candidates", "Dropped 247") are hardcoded for the
  demo query — they'd be real in production but for tonight they're scripted.
- Results come from products.json, filtered by style_tag matching the user's
  profile. The "Why we picked this" and verification check lines are stored
  in products.json directly — no live LLM generation needed for these.
- The query parser IS real — GPT-4o call to extract intent, but the result
  isn't actually used to filter; it's used only to show the parsed details
  in step 1. This is fine — investors see live AI parsing, demo runs reliably.

## Screen 10: Empty state (rendered inside /search)

Purpose: when search returns < 3 results, offer paths forward.

Layout:
- Centered, max-width 380px
- Search icon
- H2: "Only 2 picks made the cut."
- Body explaining what was filtered and why
- Three suggested adjustments with quantified impact:
  - "Allow elbow-length sleeves — would unlock ~30 more"
  - "Raise budget to $200 — ~12 more options"
  - "Show 'needs slip' matches — 8 dresses you'd love if you layer"
- Each has a "Try it" button

For the demo build: skip this screen entirely. The demo query always has
results.

---

## Screen 11: Product detail (route: /product/[id])

Purpose: verification deep-dive before user clicks out.

Layout:
- Image carousel (left), details (right)
- "Why we picked this" in serif italic
- Full verification report:
  - All standard checks with evidence
  - Up to 5 review excerpts that informed the verdict
  - Aggregate review signals (transparency, fit, quality)
- "View at [Brand]" CTA opens brand site in new tab
- Save and share buttons

For the demo build: this screen can be a simple page that shows the same
verification info from the search results card, plus a larger image. Clicking
"View at Brand" opens product_url in new tab.
